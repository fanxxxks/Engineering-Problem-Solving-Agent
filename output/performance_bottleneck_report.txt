二、并行模式分析

  当前并行策略: ThreadPoolExecutor + Semaphore(3)

  并行模式在架构上是部分有效的——3 道不同的题目可以同时被处理。但瓶颈
  在于单个题目内部的 ReAct 推理循环是严格串行的:

    ┌─ 题目 A ──────────────────────────────────────────────────┐
    │  Step 1: LLM调用 (30-90s) → 解析 → 执行工具 → 观察         │
    │  Step 2: LLM调用 (30-90s) → 解析 → 执行工具 → 观察         │
    │  Step 3: LLM调用 (30-90s) → 解析 → 执行工具 → 观察         │
    │  ...                                                        │
    │  Step N: LLM调用 → 最终答案                                 │
    │                                                             │
    │  最多 8 步 × 30-90s = 4~12 分钟/题                          │
    └─────────────────────────────────────────────────────────────┘

  ThreadPoolExecutor 只能在不同题目之间并发，不能并行化同一个题目内
  的 Think→Act→Observe 循环，因为每一步都依赖上一步的 LLM 输出结果。

  ⚠ 关键约束: Python GIL 在 I/O 等待时释放，所以 run_in_executor()
  中的 LLM HTTP 调用是真正并行的。3 路并发实际能减短总时间约 2.5x。

四、瓶颈分类与量化

  ┌───────────────────────────────────────────────────────────────────────┐
  │  瓶颈 #1: LLM API 调用耗时 (占总量 ~85%)                              │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │  • 每次 stream API 调用: 30-90 秒 (取决于输出 token 长度)             │
  │  • 单题平均 3-6 次调用，每个调用严格串行                               │
  │  • 模型: kimi-k2.5, Temperature=1.0, max_tokens 无上限                │
  │  • 高温(1.0)增加输出长度 → 有时 LLM 长篇大论（上万字思考过程）        │
  │                                                                       │
  │  恶化因素:                                                             │
  │  - LLM 惯性"过推导": 已得答案后仍额外 write 3-5 段，浪费30s+          │
  │  - 每次 "最终答案" 被 Markdown 加粗干扰后需额外 1 步纠正（已修复）    │
  │  - 证明题/分析题 LLM 倾向于输出完整教材级推导                          │
  │                                                                       │
  │  典型案例: CAL_007 (8932s) — LLM 在每一步都输出上千字详细推导，        │
  │  且频繁请求 "请继续下一步" ，导致全程 8 步，最终超时                    │
  └───────────────────────────────────────────────────────────────────────┘

  ┌───────────────────────────────────────────────────────────────────────┐
  │  瓶颈 #2: image_tool 图片传输 (占 ~10%)                               │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │  • 图片 base64 编码后约 1-5 KB，嵌入 API 请求中                       │
  │  • Vision 模型处理: 单张电路图 60-120 秒（不论大小都接近恒定）         │
  │  • 含图片题目 7 道 (CIR_004~CIR_010)，每道至少 1 次 image 调用        │
  │  • CIR_004: 图片描述 2065s (占总耗时 ~90%)                            │
  │  • CIR_005: 图片描述 231s (占总耗时 ~40%)                             │
  │                                                                       │
  │  优化空间: 图片描述 result 可缓存/复用，目前每题独立调用               │
  └───────────────────────────────────────────────────────────────────────┘

  ┌───────────────────────────────────────────────────────────────────────┐
  │  瓶颈 #3: API 过载与网络抖动 (占 ~5%)                                 │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │  • 429 Overloaded: 3 道题 (LA_007/008/009) 因 Kimi API 高峰被拒       │
  │  • brotli 解码崩溃: 6 道题因底层压缩错误导致 stream 中断               │
  │  • Timeout: 偶尔 Request timed out                                     │
  │                                                                       │
  │  两者现在都有重试机制（指数退避 + 非流式 fallback），但每个重试        │
  │  仍需等待 2-5 秒 + 1 次完整 API 调用时间                               │
  └───────────────────────────────────────────────────────────────────────┘

  ┌───────────────────────────────────────────────────────────────────────┐
  │  瓶颈 #4: 符号计算复杂度 (几乎可忽略)                                  │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │  • SymPy 本地计算: < 0.1 秒/次                                        │
  │  • 但 LLM 经常在"写代码 → 执行 → 解读结果"上浪费 1-3 步               │
  │  • compute 工具调用 → 执行 → 观察 整个循环耗时 60-120s（主要是 LLM）   │
  └───────────────────────────────────────────────────────────────────────┘
